# UIEditor_new 后续工作建议

更新时间：2026-06-25

本文是对 `UIEditor_new` 后续工作的通俗版整理。它不替代 `WORKITEM.md`、`docs/ngui-rewrite-plan.md`、`docs/ngui-interaction-todo.md` 和 `docs/unity-editor-bridge-api.md`，而是把当前剩余工作按“为什么要做、先做什么、做到什么程度算完成”重新组织，方便后续排期和交接。

## 一句话结论

`UIEditor_new` 最难的渲染链路已经过了主要关口：Unity Bridge 能打开真实 Prefab、渲染 Unity 真值截图、导出 bbox、做基础 patch、保存前做 protected diff；NGUI 的渲染污染和 bbox 同源问题也已经基本解决。

后面的重点不再是“能不能出图”，而是“能不能作为一个稳定、安全、好用的生产工具交给 UI 人员使用”。

## 当前明确还有的 TODO

### 1. 补完整 NGUI 属性栏

现状：

- Unity Bridge 侧已经能读写一部分 NGUI 字段。
- Web 前端属性面板仍主要按 UGUI 的 `Text`、`Image`、`Button` 逻辑写。
- NGUI 节点能选中、bbox 能显示、截图能渲染，但很多 NGUI 字段没有可编辑入口。

要做：

- 在节点模型里标记当前节点属于 `UGUI` 还是 `NGUI`。
- 属性面板按框架分支：UGUI 节点显示 UGUI 字段，NGUI 节点显示 NGUI 字段。
- NGUI 属性栏至少覆盖 `UILabel`、`UISprite`、`UIWidget`、`UIButton` 的常用视觉字段。

做到什么算完成：

- 选中 NGUI 文本节点，可以改文字、字号、颜色、对齐等字段。
- 选中 NGUI 图片节点，可以改 sprite、颜色、透明度、填充方式等字段。
- 选中 NGUI widget，可以改宽高、显隐、颜色、透明度等字段。
- UGUI 属性栏不被影响。

### 2. 补 NGUI 核心字段写入

现状：

文档里明确点名还有三个 NGUI 核心字段没有完整闭合：

- `Widget.depth`
- `Widget.pivot`
- `Text.overflow`

这些不是边角字段。`depth` 影响前后层级，`pivot` 影响定位和缩放基准，`overflow` 影响文字溢出方式。

要做：

- Bridge 侧支持读取和写入这些字段。
- Web 属性面板提供对应控件。
- 修改后触发 NGUI 刷新、截图回包和 Web 同步。

做到什么算完成：

- 修改 `depth` 后，NGUI 视觉层级真实变化。
- 修改 `pivot` 后，Unity 中节点 pivot 与 Web 属性显示一致。
- 修改文字 overflow 后，文字显示行为与 Unity 真实 Prefab 一致。

### 3. NGUI 画板禁止混入 UGUI 控件，或支持创建 NGUI 控件

现状：

- 画板已经有 framework 单路由：第一个非空 Prefab 决定是 UGUI 还是 NGUI。
- 但创建节点的接口仍偏 UGUI，例如创建文字会建 UGUI `Text`，创建按钮会建 UGUI `Button`。
- 如果在 NGUI 画板里建出 UGUI 控件，会破坏“一个画板只走一种 UI 框架”的规则。

要做：

优先选择一个保守方案：

- NGUI 画板上暂时禁止创建 UGUI 控件，返回明确错误。

后续再做完整方案：

- NGUI 画板创建文字时建 `UILabel`。
- NGUI 画板创建图片时建 `UISprite` 或 `UITexture`。
- NGUI 画板创建按钮时建 NGUI 风格按钮结构。

做到什么算完成：

- NGUI 画板不会被意外插入 UGUI 节点。
- 用户看到明确提示，而不是创建后画面或保存行为变异常。

### 4. 拖动过程增加 Unity 真值节流回填

现状：

- 拖动时 Web 端先用本地叠加层即时反馈，所以手感已经比较快。
- 松手后 Unity 才返回真值截图。
- 拖动过程中没有持续用 Unity 真值校正，复杂 NGUI 场景下容易出现拖动时看起来顺，松手后跳一下。

要做：

- 拖动时每隔约 120 到 150ms 向 Unity 发一次低频 move 请求并请求截图。
- 新请求发出前取消旧请求，避免旧截图晚回来覆盖新状态。
- 给请求加递增序号，旧回包直接丢弃。
- 松手后再发最终一次确认。

做到什么算完成：

- 拖动过程仍然跟手。
- 松手后节点不会明显跳变。
- 网络或 Unity 主线程慢时，旧截图不会覆盖新截图。

### 5. protected diff 升级到字段级和组件级报告

现状：

- protected diff 已经从脆弱的 YAML 逐行 diff 升级为 MVP 结构签名校验。
- 现在能避免 Prefab Variant modification 重排造成的大量误报。
- 但报告还不够细，不能清晰告诉用户“哪个节点的哪个字段变了”。

要做：

- 输出字段级 allowed changes。
- 输出字段级 protected changes。
- 输出组件级变化，例如组件新增、删除、脚本引用变化、事件绑定变化。
- Web 保存前展示清晰摘要。

做到什么算完成：

- 保存前能看到允许变化清单。
- 如果保存被拒绝，能定位到具体节点、组件、字段和原因。
- UI 人员不需要读 YAML 也能理解为什么不能保存。

### 6. 补只读字段和不可改原因提示

现状：

- 后端已有白名单和保护字段概念。
- Web 上还缺少清楚提示。
- 用户看到字段不可改时，不知道是没做完、没有选中、还是出于安全限制。

要做：

- 每个不可编辑字段显示原因。
- 常见原因包括：脚本绑定、事件接线、Lua/schema 绑定、Prefab source、非视觉组件、当前框架不支持。
- 属性栏中只读项和可编辑项视觉上区分清楚。

做到什么算完成：

- 用户知道哪些字段能改。
- 用户知道哪些字段不能改，以及为什么不能改。
- protected diff 拦截前，前端就尽量避免用户提交危险操作。

### 7. 补面向 UI 人员的短使用说明

现状：

- 现在文档偏工程交接和研发视角。
- 对 UI 人员来说，还缺一份很短的“怎么用、哪些能改、出错看哪里”的说明。

要做：

- 写一份短使用说明。
- 内容包括：启动、打开画板、选择节点、拖动、改属性、保存、关闭、遇到 Bridge 失败怎么办。
- 明确说明：工具只允许改视觉字段，不允许改脚本、事件、绑定和 Prefab 结构风险项。

做到什么算完成：

- 新接手的人不需要翻完整技术文档，也能跑通一次低风险样本。
